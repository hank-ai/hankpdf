# v0.3.0 Mega-Release — Design Spec

**Status:** brainstormed, awaiting user review
**Date:** 2026-05-02
**Author:** Jack Neil + Claude
**Target version:** `hankpdf` 0.3.0
**Predecessor:** 0.2.2 (CTO terminology refresh, 2026-04-29)

---

## 1. Goal

Cut a single, coherent 0.3.0 release that delivers:

1. **Production-safety hardening** — per-page worker memory caps on Linux, macOS, and Windows; a decompression-bomb regression corpus; refusal-by-default for digitally signed PDFs.
2. **Clean breaking changes** — drop the `pdf_smasher/` deprecation shim that was promised in the 0.2.0 CHANGELOG.
3. **Better fresh-machine UX** — fail loud with install hints when native dependencies are missing or out of date, instead of crashing deep in a stack trace.
4. **Small API debt cleanup** — `correlation_id` kwarg on `compress_stream()`, ROADMAP terminology fix, and a localized refactor that pulls the per-page MRC gate out of the 1455-line `__init__.py`.

The release is intentionally bigger than a routine bump because the items are interdependent (signed-PDF refusal, memory caps, and bomb corpus together form the safety story; shim removal and CHANGELOG migration share release notes; doctor output spans both UX and version-floor enforcement). Pre-1.0 SemVer permits the breaks.

## 2. Non-goals

- Per-mode threshold tuning (Roadmap T2.6) — out of scope; needs Phase 8 corpus expansion first.
- v0.3.0 does **not** rename `mode=medical` in code or CLI; only updates the unfinished ROADMAP rows. The actual third mode is not implemented in 0.2.x and stays unimplemented in 0.3.0.
- Phase 8 dogfood corpus expansion (separate work).
- Triaging the 7 open dependabot PRs (per-PR review, separate work).
- Manual `pdf-smasher 0.1.0` PyPI yank (user-only action).

## 4. Detailed design

### A. Drop `pdf_smasher/` deprecation shim

**Today:** `pdf_smasher/__init__.py` is a 30-line shim that re-exports everything from `hankpdf` and emits `DeprecationWarning`. It was added in 0.2.0 and CHANGELOG promised removal in 0.3.0.

**Change:** delete the `pdf_smasher/` directory entirely. Remove the `pdf_smasher` entry from `[tool.setuptools.packages.find]` (or equivalent) in `pyproject.toml`. The wheel ships with only `hankpdf/` going forward.

**Migration note (CHANGELOG):**

```
BREAKING: Removed pdf_smasher/ deprecation shim.
  - Was: from pdf_smasher import compress
  - Now: from hankpdf import compress
  - Was: pdf_smasher.__version__
  - Now: hankpdf.__version__
  - Was: importlib.metadata.version("pdf-smasher")
  - Now: importlib.metadata.version("hankpdf")
  Last warning was in 0.2.x. If you missed the warning window, run
  your test suite under PYTHONWARNINGS=error::DeprecationWarning to
  catch any remaining shim usage.
```

**Tests:** delete `tests/integration/test_install_smoke.py` cases that asserted shim warning; keep clean-venv canonical-import case (rename to drop "shim" terminology).

### B. Lazy native-dep boot check

**Today:** `--doctor` exists at `hankpdf/cli/main.py:564` (`_doctor_report`) and prints versions. It is informational only. If a user runs `hankpdf in.pdf` without Tesseract installed, the failure surfaces deep inside `pytesseract.image_to_pdf_or_hocr()` as a `TesseractNotFoundError` with no install hint.

**Change:** new module `hankpdf/_environment.py` exporting:

```python
@dataclass(frozen=True, slots=True)
class EnvironmentReport:
    python_version: str
    platform: str
    tesseract: tuple[str, str | None] | None  # (path, version) or None if missing
    qpdf: tuple[str, str | None] | None
    jbig2enc: tuple[str, str | None] | None
    openjpeg_via_pillow: str | None
    pdfium_revision: str | None
    pillow_max_image_pixels: int
    failures: tuple[EnvFailure, ...]

@dataclass(frozen=True, slots=True)
class EnvFailure:
    component: str   # "tesseract" | "qpdf" | "jbig2enc" | "openjpeg" | ...
    reason: str      # "missing" | "below-floor" | "unparseable-version" | ...
    found: str | None
    required: str | None
    install_hint: str  # platform-specific one-liner

class MissingDependencyError(RuntimeError):
    """Raised by assert_environment_ready when one or more required
    native dependencies is missing or below its supported floor.
    """
    failures: tuple[EnvFailure, ...]

def assert_environment_ready() -> EnvironmentReport: ...
def get_environment_report() -> EnvironmentReport: ...  # cached
```

`get_environment_report()` is cached for the process lifetime via `functools.cache`. First call performs every `subprocess.run([tool, "--version"], timeout=5)`. Subsequent calls return the cached `EnvironmentReport`.

`assert_environment_ready()` calls `get_environment_report()` and raises `MissingDependencyError` if any `EnvFailure` is present, formatted as:

```
hankpdf cannot start. Required native dependencies are missing or below the supported floor:

  tesseract  : not found in PATH
               install: brew install tesseract  (macOS)
                        sudo apt install tesseract-ocr  (Debian/Ubuntu)
                        choco install tesseract  (Windows + Chocolatey)

  qpdf       : 11.3.0 found, 11.6.3 required (silent character-drop bug below 11.6.3)
               install: brew upgrade qpdf  (macOS)
                        Trixie or backport: see docs/INSTALL.md

Run `hankpdf --doctor` for the full report.
```

Exit code: **41** (`E-ENV-MISSING`) — already reserved for environment errors per SPEC §2.2.

**Hook points:**

- `hankpdf.compress()` and `hankpdf.compress_stream()`: call `assert_environment_ready()` before any other work (after argument validation). Library users only pay the subprocess cost when they actually call compress.
- `hankpdf.cli.main.main()`: call `assert_environment_ready()` immediately after argparse (before reading input bytes). Catches the "missing tesseract" failure before any I/O.
- `hankpdf.cli.main._doctor_report()`: rewrite to consume `get_environment_report()` so doctor output and boot-check output are byte-aligned.

`MissingDependencyError` is caught at the CLI boundary in `cli/main.py` and translated into exit code 41 with the formatted multi-line message printed to stderr. Library callers see the exception directly with `failures` attached for programmatic handling.

**Floors enforced (per Roadmap T0.9):**

| Component       | Minimum  | Rationale                                                                                 |
| --------------- | -------- | ----------------------------------------------------------------------------------------- |
| Python          | 3.14     | Already enforced via `requires-python` in `pyproject.toml`.                              |
| Tesseract       | 5.0.0    | tessdata format; required by current OCR pipeline.                                        |
| qpdf            | 11.6.3   | Silent character-drop bug in 11.0–11.6.2 (qpdf #1050 / Launchpad #2039804).               |
| OpenJPEG        | 2.5.4    | CVE-2025-54874 OOB heap write in ≤2.5.3. Probed via Pillow's bundled feature reporting.   |
| jbig2enc        | present  | Vendored commit recorded; absence drops text-only ratio ~50× → ~6× (warning, not error). |

**Library opt-out:** environment variable `HANKPDF_SKIP_ENV_CHECK=1` skips the check entirely. Used by tests that intentionally run with a stub Tesseract. Documented in `docs/ENVIRONMENT.md`.

### C. `correlation_id` kwarg on `compress_stream()`

**Today:** `compress_stream()` (`hankpdf/__init__.py:1446`) generates a fresh UUID4 per call. Programmatic callers that already track a correlation ID upstream can't tie hankpdf's stderr / sidecar manifest back to their own logger.

**Change:** add `correlation_id: str | None = None` kwarg to:

- `hankpdf.compress(...)`
- `hankpdf.compress_stream(...)`

Behavior: if `correlation_id is None`, generate `uuid.uuid4().hex` (current behavior). If a string is passed, validate via `re.fullmatch(r"[A-Za-z0-9._:-]{1,64}", correlation_id)`; reject with `ValueError("correlation_id must match [A-Za-z0-9._:-]{1,64}")`.

Validation rationale: stays printable, no whitespace (parses cleanly out of stderr lines), no control characters, length-bounded (avoid log-injection abuse).

The CompressReport `correlation_id` field already exists; nothing changes downstream.

### D. Per-page worker memory caps (Linux, macOS, Windows)

**Today:** `hankpdf/sandbox/__init__.py` is an 8-line stub promising RLIMIT + Job Object support that was never implemented. Per-page workers run with whatever the host's process default is. A pathological PDF (1.2 billion-pixel page, infinite xref loop, declared `/Length` 100× actual) can OOM the host before any timeout fires.

**Change:** new module `hankpdf/sandbox/platform_caps.py` with one public surface:

```python
def apply_self_memory_cap(byte_limit: int) -> None:
    """Apply a per-process memory cap to the *current* process.

    Called from inside ProcessPoolExecutor workers via _init_worker.
    Raises CapsUnavailableError with a structured reason if no cap
    primitive is available on this OS.
    """
```

**Linux/macOS implementation** (`_apply_unix.py`):

```python
import resource
def _apply(byte_limit: int) -> None:
    resource.setrlimit(resource.RLIMIT_AS, (byte_limit, byte_limit))
```

`RLIMIT_AS` (address space) is honored on both Linux and macOS. `RLIMIT_DATA` is too narrow on Linux (mmap-backed mallocs bypass it); `RLIMIT_RSS` is no-op on Linux. `RLIMIT_AS` it is, with a generous cap (see "Threshold" below).

**Windows implementation** (`_apply_win32.py`):

```python
import ctypes
import ctypes.wintypes as wt

# kernel32 types
class JOBOBJECT_BASIC_LIMIT_INFORMATION(ctypes.Structure):
    _fields_ = [
        ("PerProcessUserTimeLimit", ctypes.c_int64),
        ("PerJobUserTimeLimit", ctypes.c_int64),
        ("LimitFlags", wt.DWORD),
        ("MinimumWorkingSetSize", ctypes.c_size_t),
        ("MaximumWorkingSetSize", ctypes.c_size_t),
        ("ActiveProcessLimit", wt.DWORD),
        ("Affinity", ctypes.c_void_p),
        ("PriorityClass", wt.DWORD),
        ("SchedulingClass", wt.DWORD),
    ]

class IO_COUNTERS(ctypes.Structure):
    _fields_ = [
        ("ReadOperationCount", ctypes.c_uint64),
        ("WriteOperationCount", ctypes.c_uint64),
        ("OtherOperationCount", ctypes.c_uint64),
        ("ReadTransferCount", ctypes.c_uint64),
        ("WriteTransferCount", ctypes.c_uint64),
        ("OtherTransferCount", ctypes.c_uint64),
    ]

class JOBOBJECT_EXTENDED_LIMIT_INFORMATION(ctypes.Structure):
    _fields_ = [
        ("BasicLimitInformation", JOBOBJECT_BASIC_LIMIT_INFORMATION),
        ("IoInfo", IO_COUNTERS),
        ("ProcessMemoryLimit", ctypes.c_size_t),
        ("JobMemoryLimit", ctypes.c_size_t),
        ("PeakProcessMemoryUsed", ctypes.c_size_t),
        ("PeakJobMemoryUsed", ctypes.c_size_t),
    ]

JOB_OBJECT_LIMIT_PROCESS_MEMORY = 0x100
JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE = 0x2000
JobObjectExtendedLimitInformation = 9

def _apply(byte_limit: int) -> None:
    kernel32 = ctypes.WinDLL("kernel32", use_last_error=True)
    kernel32.CreateJobObjectW.restype = wt.HANDLE
    kernel32.CreateJobObjectW.argtypes = [ctypes.c_void_p, wt.LPCWSTR]
    h_job = kernel32.CreateJobObjectW(None, None)
    if not h_job:
        raise OSError(ctypes.get_last_error(), "CreateJobObjectW")

    info = JOBOBJECT_EXTENDED_LIMIT_INFORMATION()
    info.BasicLimitInformation.LimitFlags = (
        JOB_OBJECT_LIMIT_PROCESS_MEMORY | JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE
    )
    info.ProcessMemoryLimit = byte_limit

    kernel32.SetInformationJobObject.restype = wt.BOOL
    kernel32.SetInformationJobObject.argtypes = [
        wt.HANDLE, ctypes.c_int, ctypes.c_void_p, wt.DWORD
    ]
    if not kernel32.SetInformationJobObject(
        h_job,
        JobObjectExtendedLimitInformation,
        ctypes.byref(info),
        ctypes.sizeof(info),
    ):
        raise OSError(ctypes.get_last_error(), "SetInformationJobObject")

    kernel32.AssignProcessToJobObject.restype = wt.BOOL
    kernel32.AssignProcessToJobObject.argtypes = [wt.HANDLE, wt.HANDLE]
    kernel32.GetCurrentProcess.restype = wt.HANDLE
    kernel32.GetCurrentProcess.argtypes = []
    h_proc = kernel32.GetCurrentProcess()
    if not kernel32.AssignProcessToJobObject(h_job, h_proc):
        raise OSError(ctypes.get_last_error(), "AssignProcessToJobObject")
    # Intentionally leak h_job: closing it would kill the worker due to
    # KILL_ON_JOB_CLOSE. The handle is released by Windows when the
    # worker process exits.
```

Public surface of `hankpdf/sandbox/platform_caps.py`:

```python
class CapsUnavailableError(RuntimeError):
    """Raised when no memory-cap primitive is available on this OS."""

def apply_self_memory_cap(byte_limit: int) -> None: ...
def supported() -> bool: ...
```

`supported()` returns `True` when `apply_self_memory_cap` will succeed on the current platform. The cap value is internally clamped to `min(byte_limit, sys.maxsize)` to fit `c_size_t` on 32-bit Python builds (theoretical only — Python 3.14 dropped 32-bit Windows wheels).

Self-assignment to a Job Object is supported on Windows 8 and later (nested jobs). hankpdf already requires Python 3.14, which dropped support for Windows 7/8 upstream — this is a non-issue.

**Hook point:** `_init_worker()` at `hankpdf/__init__.py:126`. Already runs as ProcessPoolExecutor's `initializer` callback inside each worker process.

Today the signature is `def _init_worker() -> None:` and `initargs=()`. This release changes it to `def _init_worker(mem_cap: int) -> None:` (positional, matching multiprocessing initargs convention) and the executor invocation to `initargs=(mem_cap,)`:

```python
def _init_worker(mem_cap: int) -> None:
    # ... existing single-thread-pin logic ...
    if mem_cap > 0:
        from hankpdf.sandbox.platform_caps import apply_self_memory_cap
        try:
            apply_self_memory_cap(mem_cap)
        except CapsUnavailableError as e:
            # caps unavailable on this platform — log and continue;
            # other layers (timeout, Pillow guards) still apply.
            sys.stderr.write(f"[W-CAPS-UNAVAILABLE] {e}\n")
```

`mem_cap=0` disables the cap (used by tests that need to allocate beyond the cap intentionally).

**Threshold** (default cap):

```python
mem_cap = max(8 * 1024**3, 16 * input_file_size_bytes)
# floor: 8 GB
# scales with input size for very large PDFs (a 200 MB scan can map 4-8 GB
# legitimately during 300 DPI rasterization)
```

This is generous on purpose. The cap is a backstop against pathological bombs (gigabyte-scale spurious allocations), not a tight memory governor. CLI flag `--max-worker-memory-mb` overrides.

**RSS watchdog (parent-side backstop):** in `compress()`'s `ProcessPoolExecutor` driver, spawn a `threading.Thread` that uses `psutil.Process(pid).memory_info().rss` to poll each live worker every 500 ms. If RSS exceeds the same cap, kill the worker via `proc.terminate()` and surface as `MemoryCapExceededError` with structured warning code `[W-MEM-RSS-WATCHDOG]`.

The watchdog catches Linux malloc-by-mmap escapes and validates Windows Job Object enforcement. Cost: one Thread + one psutil call per 500 ms per worker. Negligible.

**Tests:** new file `tests/integration/test_memory_caps.py`:

- "spend 10 GB" worker function as a fixture; expect `MemoryCapExceededError` on all three OSes in CI.
- Verify cap is applied correctly by reading `psutil.Process().memory_maps()` (Linux/macOS) or `GetProcessMemoryInfo` (Windows) inside the worker.

### E. Signed-PDF default refuse

**Today:**

- `hankpdf/engine/triage.py:152` — `_detect_signature()` returns `(is_signed: bool, is_certified: bool)`.
- `hankpdf/__init__.py:266` — refuses certifying signatures unless `options.allow_certified_invalidation` is set, with `E-INPUT-CERTIFIED`.
- Non-certifying signatures: silently invalidated. Compressor rewrites the file; any embedded signature becomes invalid because the byte range hash no longer matches. **This is a real bug**, not just incomplete UX.

**Change:** introduce three states for signed-PDF handling, gated by two new options/flags:

```python
@dataclass
class CompressOptions:
    # ... existing ...
    allow_signed_invalidation: bool = False
    preserve_signatures: bool = False
```

Engine logic post-triage (after the existing certifying-signature check):

```python
if tri.is_signed and not tri.is_certified_signature:
    if options.preserve_signatures:
        # Pass-through verbatim; record reason in CompressReport.
        return _passthrough_compress_report(
            input_bytes,
            reason="passthrough-signed",
            warnings=("[W-PASSTHROUGH-SIGNED] input has digital signature; "
                      "passthrough preserves the signature",),
        )
    if not options.allow_signed_invalidation:
        raise SignedPDFError(
            "input has a digital signature; recompression would invalidate it. "
            "Pass --preserve-signatures to passthrough, or "
            "--allow-signed-invalidation to recompress and break the signature."
        )
    # Else: fall through to the normal pipeline, sig will be invalidated.
    # Record signature_invalidated=True in the report.
```

Mutual exclusion: `--preserve-signatures` and `--allow-signed-invalidation` are mutually exclusive. CLI argparse rejects the combination with exit 2 and a clear message. Library callers raise `ValueError`.

Same model for the CLI flag pair (`hankpdf/cli/main.py`):

```python
sig_group = p.add_mutually_exclusive_group()
sig_group.add_argument("--preserve-signatures", action="store_true",
    help="When input has a digital signature, copy bytes verbatim "
         "(no compression) instead of refusing.")
sig_group.add_argument("--allow-signed-invalidation", action="store_true",
    help="When input has a digital signature, recompress anyway. "
         "The signature will become invalid.")
```

`SignedPDFError` already exists at `hankpdf/exceptions.py:20`. Wire its handler in CLI to exit code **42** (`E-INPUT-SIGNED`) — already documented in SPEC §8.5 warning codes. Add to changelog.

**CompressReport fields (additive, schema bumps v4 → v5):**

```python
@dataclass(frozen=True, slots=True)

    # ... existing ...
    schema_version: int = 5
    signature_state: Literal["none", "passthrough-preserved",
                             "invalidated-allowed", "certified-invalidated-allowed"]

```

### H. Extract `_run_per_page_gate()` from `compress()`

**Today:** `hankpdf/__init__.py:789-1455` is a 660-line `compress()` function. The per-page selective MRC gate (~130 lines: `_score_pages_for_mrc` call, threshold logic, `mrc_worthy` flag injection into `_WorkerInput`, passthrough fast-path) lives inline.

**Change:** move that block into a new file `hankpdf/engine/per_page_gate.py`:

```python
def run_per_page_gate(
    pdf: pikepdf.Pdf,
    triage: TriageReport,
    options: CompressOptions,
) -> PerPageGateResult:
    """Score pages, decide passthrough vs MRC, return gate result.

    Returns:
        PerPageGateResult with mrc_worthy: tuple[bool, ...] and
        whole_doc_passthrough: bool. Caller threads mrc_worthy into
        _WorkerInput; if whole_doc_passthrough, caller emits
        passthrough-no-image-content warning and returns input bytes
        unchanged.
    """
```

`PerPageGateResult` is a frozen dataclass. The caller in `compress()` becomes:

```python
gate = run_per_page_gate(pdf, tri, options)
if gate.whole_doc_passthrough:
    return _passthrough_compress_report(...)
# else: thread gate.mrc_worthy into _WorkerInput per page
```

No behavior change. Tests for the gate (already present in `tests/unit/test_compress_per_page_gate.py`) get retargeted to import from `hankpdf.engine.per_page_gate`. CHANGELOG: "Internal refactor only; no behavior change."

This is a non-blocking refactor that pays off the Wave-1 DCR finding "compress() is too long". Done in this PR because we're already touching that code path for items B, D, E.

## 5. Data model changes

### CompressOptions (existing dataclass in `hankpdf/types.py` — ADD fields)

```python
# Add to existing CompressOptions:
allow_signed_invalidation: bool = False
preserve_signatures: bool = False
max_worker_memory_mb: int | None = None  # None = compute from input size
correlation_id: str | None = None
```

Validation (`__post_init__`):

- `allow_signed_invalidation` and `preserve_signatures` are mutually exclusive (raise `ValueError`).
- `allow_signed_invalidation` is **orthogonal** to `allow_certified_invalidation`. They cover different signature classes (non-certifying vs certifying) and may both be set.
- `correlation_id`: regex `[A-Za-z0-9._:-]{1,64}`.
- `max_worker_memory_mb`: positive int, max `1_000_000` (1 TB sanity cap).

### CompressReport (schema v4 → v5)

Add fields:

```python
schema_version: int = 5
signature_state: Literal["none", "passthrough-preserved",
                         "invalidated-allowed", "certified-invalidated-allowed"]
signature_invalidated: bool
worker_memory_cap_bytes: int  # the cap that was applied
worker_peak_rss_max_bytes: int  # max RSS observed across all workers
```

Existing 0.2.x consumers reading the report by key name continue to work; only new fields added.

## 8. Known limitations / explicitly out of scope

- **Signed-PDF preservation via incremental update is impossible for compression.** The PDF spec only permits incremental updates to APPEND data, not to rewrite existing image streams to JBIG2/JPEG2000. The "preserve" path in this design is verbatim passthrough (no compression). The Roadmap T2.14 phrasing "preserve via incremental update" is corrected during this release — incremental update preserves *modifications* (annotations, form fills) on signed PDFs, not compression.
- **`mode=strict` itself is not implemented.** Only the docs are renamed. Implementing the third mode is Phase-2 follow-on work (T2.3, T2.6).
- **Windows memory caps require Windows 8 or later.** Python 3.14 already drops support for Win 7/8, so this matches the supported platform floor.
- **Worker memory cap floor is 8 GB.** Tighter caps would surface false-positive OOM on legitimate large 300 DPI rasterization. The cap is a backstop against pathological bombs, not a tight memory governor.
- **psutil RSS watchdog runs at 500 ms cadence.** A spike that grows from below-cap to OOM in less than 500 ms could escape the watchdog. The kernel-level RLIMIT_AS / Job Object cap is the authoritative enforcement; the watchdog is a redundant backstop.
- **One new Python runtime dependency: `psutil>=5.9`** for the parent-side RSS watchdog. Cross-platform, MIT-licensed, ~280M monthly downloads, ~50 KB wheel. Considered alternatives: per-platform ctypes (`/proc/<pid>/statm` Linux + Mach `task_info` macOS + `GetProcessMemoryInfo` Windows). Rejected because the watchdog is a redundant backstop already; adding 3 platforms of bespoke ctypes for redundancy is worse than one well-tested wheel. Job Object support itself does NOT need psutil — that's pure ctypes against `kernel32.dll`. No `pywin32` dep.
- **One new test-only dependency: `pyhanko>=0.27`** to generate signed-PDF fixtures in `scripts/generate_signed_corpus.py`. Added under `[project.optional-dependencies] test`, not the default install.
- **The 7 open dependabot PRs and the manual `pdf-smasher 0.1.0` PyPI yank are out of scope** for this release.
